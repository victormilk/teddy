# Phase 07 Plan 01: Skill Flows com Auto-Reconhecimento

**Implementar /teddy:flows — sistema de skill dependencies com auto-discovery, matching semântico, e confirmation flow. Integrar no loop existente: detecção no PLAN, gate no APPLY, audit no UNIFY.**

## Objective

- **Goal:** Novo comando /teddy:flows com 4 subcommands (config, add, audit, list), auto-discovery de skills com leitura de frontmatter, matching por contexto, confirmation flow interativo. Integração no plan.md (detect), apply.md (verify/block), unify.md (audit). Framework reference + template.
- **Purpose:** Transformar skill management de "manual e esquecível" para "proativo e confirmado" — diferencial vs Paul que usa apenas mapeamento manual
- **Output:** 3 novos arquivos (flows.md, skill-flows.md, FLOWS.md template), 5 arquivos existentes modificados (plan, apply, unify, plan-format, loop-phases), entry point e docs atualizados, version bump para 0.6.0

    ## Purpose
    Define how Teddy discovers, matches, and enforces skill dependencies across the development loop. The key insight: skills that are important enough to use are important enough to track — but tracking should be proactive, not manual.

    ## The 3-Layer Model

    ```
    DISCOVERY (what exists?)
      └─ Scan ~/.claude/commands/ recursively
      └─ Read frontmatter: description, name, when-to-use
      └─ Categorize: design, backend, frontend, testing, devops, content, security

    MATCHING (what is relevant?)
      └─ Cross PROJECT.md (stack, constraints) with skill metadata
      └─ Cross ROADMAP.md (phase goals) with skill triggers
      └─ Cross PLAN.md tasks with skill when-to-use
      └─ Generate suggestions with confidence: HIGH / MEDIUM / LOW

    CONFIRMATION (user validates)
      └─ Present suggestions with match reasons
      └─ User accepts / rejects / adjusts each suggestion
      └─ Save to .teddy/FLOWS.md with metadata
      └─ Rejected skills marked to avoid re-suggesting
    ```

    ## Priority Levels

    | Priority | PLAN Behavior | APPLY Behavior | UNIFY Behavior |
    |----------|---------------|----------------|----------------|
    | required | Added to <skills> section | Blocks execution if not loaded | Gap documented in SUMMARY.md |
    | optional | Listed in <skills> section | No blocking | Informational only |

    ## Enforcement Hierarchy

    Document the integration points:

    ```
    /teddy:flows     → "This project uses X, Y, Z skills" (FLOWS.md)
         │
    /teddy:plan      → "This plan needs X skill" (<skills> section, detect step)
         │
    /teddy:apply     → "Is X loaded?" (blocking gate for required skills)
         │
    /teddy:unify     → "Was X actually used?" (audit, warn don't block)
    ```

    ### At PLAN (detect_required_skills step)
    - Reads FLOWS.md for declared skills
    - Analyzes plan tasks against skill metadata
    - Suggests additional skills based on task content
    - User confirms → added to PLAN.md <skills> section
    - Runs BETWEEN design_team_composition and create_plan

    ### At APPLY (verify_required_skills step)
    - Reads PLAN.md <skills> section
    - For each required skill: checks if loaded in session
    - If missing: BLOCKS with clear message listing missing skills
    - User can: "ready" (re-check) or "override" (logged to STATE.md)
    - Priority: blocking — runs AFTER validate_approval, BEFORE load_plan

    ### At UNIFY (audit_skill_usage step)
    - Cross-references FLOWS.md + PLAN.md <skills> with execution
    - Gaps documented in SUMMARY.md Deviations section
    - Philosophy: **warn, don't block** — UNIFY always completes
    - Runs AFTER compare_plan_vs_actual, BEFORE create_summary

    ## FLOWS.md Format

    Document the template structure:
    - Discovered Skills: all skills found during scan
    - Project Mappings: confirmed skill → work type mappings with metadata
    - Rejected Suggestions: skills user explicitly declined (with reason and date)
    - Phase Overrides: phase-specific skill additions
    - Amendment History: configuration change log

    ## The <skills> Section in PLAN.md

    ```xml
    <skills>
    | Skill | Priority | Reason |
    |-------|----------|--------|
    | /frontend-design | required | Plan has 8 UI component tasks |
    | /api-patterns | optional | API routes in scope |
    </skills>
    ```

    This section is populated by detect_required_skills during PLAN and verified by APPLY.

    ## Confidence Levels

    | Level | Meaning | How Determined |
    |-------|---------|----------------|
    | HIGH | Direct keyword match between skill and project/phase | Stack technology matches skill description |
    | MEDIUM | Category match without direct keywords | Skill category aligns with work domains |
    | LOW | Indirect or speculative match | Weak association, user should evaluate |

    ## Rejected Suggestions

    When a user rejects a suggested skill:
    - Reason is recorded in FLOWS.md Rejected Suggestions table
    - Skill will NOT be re-suggested in future /teddy:flows runs for this project
    - User can manually re-add via /teddy:flows add if they change their mind

    ## Anti-Patterns

    | Anti-Pattern | Fix |
    |-------------|-----|
    | Blocking UNIFY on skill gaps | Warn and document, never block — UNIFY must complete |
    | Suggesting without confidence | Always include match reason and confidence level |
    | Re-suggesting rejected skills | Check Rejected Suggestions before matching |
    | Marking everything required | Required = team quality depends on it. Optional = nice-to-have |
    | Ignoring audit results | Review UNIFY audit to update FLOWS.md over time |
  </action>
